일하는 방식

“어렵다”고 말하기 전에,
대안을 들고 갑니다.

렌더링 부하가 큰 시안, 어떻게 했을까

26종 차트 개발 중, 검토 없이 확정된 반원 차트 시안이 있었습니다. 데이터 값마다 꼬리 달린 말풍선 라벨을 호 위에 배치하는 구조였는데, 값에 따라 좌표·꼬리 각도를 동적으로 계산해야 해서 리플로우 비용이 크다고 판단했습니다.

단순히 “어렵다”고 전달하는 대신, 동적 좌표·각도 계산을 피하는 방향으로 대안 2종을 직접 구현해 가져갔습니다. 라벨을 호 안쪽에 고정 배치하는 방식, 상세 정보를 호버 툴팁으로 옮겨 상시 렌더링 요소를 줄이는 방식. 고객사 재협의 끝에 제안한 형태 중 하나로 스펙이 바뀌었습니다.

버튼이 아니라 토글이어야 하는 이유

프롬프트 입력 ON/OFF 기능이 일반 버튼 형태로 구현된 시안을 검토하던 중, 해당 UI가 컴포넌트의 역할과 맞지 않다고 판단했습니다. 버튼은 “동작이 실행된다”는 의미를, 토글은 “상태를 전환한다”는 의미를 전달하기 때문입니다.

별도 요청 없이 토글로의 변경을 제안했고, 고객사에서도 “사용성도 좋아지고 눈에도 잘 띈다”며 먼저 의견을 제시해줘서 고맙다는 반응이었습니다.

불편해도 TypeScript, 왜 밀어붙였나

SI 특성상 빠른 개발이 우선시되어 팀 내 TypeScript 도입 시도 자체가 없었습니다. 이전 이커머스 프로젝트에서 숫자가 문자열로 넘어와 서비스 오류가 발생한 사례를 들어 팀을 설득했고, “조금 불편하더라도 서비스는 안정적이어야 한다”는 논리로 도입을 추진했습니다. 이후 LG전자 리테일 관리 시스템과 HSAD AI Agent 두 프로젝트 모두 TypeScript 기반으로 진행되었습니다.

외부 API가 이상할 때, 우리 코드만 고쳐야 할까

HD현대 채용 플랫폼에서 외부 채용공고 플랫폼(잡다)의 API 응답이 구·신버전 타입으로 혼재되어 있었지만 구분값이 없는 문제를 발견했습니다. 프론트에서 억지로 파싱할 수도 있었지만, 근본 원인은 API 응답 설계에 있다고 판단했습니다.

직접 잡다 측에 구분값 추가를 요청했고, “Base URL은 저희 쪽에서 맞춰 호출하겠다”는 조건을 함께 제시해 상대방이 받아들이기 쉽게 했습니다. 협의 끝에 구분값이 추가되어 분기 처리 구조로 깔끔하게 대응할 수 있었습니다. 프론트에서 우회하기보다, 원인이 있는 곳에서 해결하는 편이 낫다고 판단했습니다.
